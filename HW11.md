a. Why Spring Boot? Pros and cons
Spring Boot is a framework built on top of Spring that removes most of the boilerplate configuration. The main advantage is auto-configuration — it detects what's on the classpath and configures things automatically, so I can start a production-ready web server with almost no setup. It also has an embedded Tomcat so I don't need to deploy a WAR file to an external server, just run the jar. Starter dependencies bundle everything I need for a feature into one import. The tradeoff is it can feel like magic — when something goes wrong it's harder to debug because so much is happening under the hood. It also pulls in a lot of dependencies which increases the build size. But for most backend projects the productivity gain far outweighs those downsides.

b. How to start a Spring Boot project from scratch
Go to start.spring.io, select Maven, Java, and the Spring Boot version, add the dependencies I need — Spring Web, Spring Data JPA, PostgreSQL Driver. Download and unzip, open in IntelliJ. The entry point is the class annotated with @SpringBootApplication which combines @Configuration, @EnableAutoConfiguration, and @ComponentScan. From there I create my layers — Controller, Service, Repository, and Entity. Add database config to application.properties and run.

c. @Controller vs @RestController
@Controller is the base annotation — it marks a class as a Spring MVC controller but by default returns view names, meaning it expects a template engine like Thymeleaf to render HTML. @RestController is @Controller plus @ResponseBody on every method, meaning every method automatically serializes the return value to JSON and writes it directly to the HTTP response. For REST APIs I always use @RestController. @Controller is for server-side rendered web apps.

d. @PathVariable vs @RequestParam
Both extract data from the URL but from different parts. @PathVariable pulls from the URL path itself — like /employees/{id} where id is embedded in the path. @RequestParam pulls from the query string — like /employees?department=engineering. I use @PathVariable when the value identifies a specific resource, @RequestParam for optional filters or search parameters.
java@GetMapping("/employees/{id}")
public Employee getById(@PathVariable Long id) {}

@GetMapping("/employees")
public List<Employee> getByDept(@RequestParam String department) {}

e. @RequestBody vs @ResponseBody
@RequestBody tells Spring to deserialize the incoming HTTP request body from JSON into a Java object. @ResponseBody tells Spring to serialize the return value of a method into JSON and write it to the HTTP response. When I use @RestController, @ResponseBody is applied automatically to every method so I don't need to add it manually.
java@PostMapping("/employees")
public Employee create(@RequestBody Employee employee) {}
// @ResponseBody implicit via @RestController

f. GetMapping, PostMapping, PutMapping, DeleteMapping, RequestMapping
@RequestMapping is the generic annotation — I put it on the class to set the base path, and it can also be used on methods with a method attribute. The others are shortcuts for specific HTTP methods. @GetMapping for reads, @PostMapping for creates, @PutMapping for full replace, @DeleteMapping for deletes. @PatchMapping exists too for partial updates.
java@RestController
@RequestMapping("/employees")
public class EmployeeController {
    @GetMapping("/{id}")        // GET /employees/{id}
    @PostMapping                // POST /employees
    @PutMapping("/{id}")        // PUT /employees/{id}
    @DeleteMapping("/{id}")     // DELETE /employees/{id}
}

g. What is Spring Actuator
Spring Actuator exposes built-in endpoints for monitoring and managing a running Spring Boot application. Out of the box I get endpoints like /actuator/health to check if the app is up, /actuator/metrics for JVM and application metrics, /actuator/env for environment properties, and /actuator/info for custom app info. It integrates with Prometheus — I add the micrometer-registry-prometheus dependency and /actuator/prometheus exposes metrics in a format Prometheus can scrape. Grafana then connects to Prometheus as a data source and visualizes the metrics as dashboards.

h. How to achieve async in Spring Boot
Add @EnableAsync on the main class or a config class, then annotate any method I want to run asynchronously with @Async. When that method is called Spring runs it on a separate thread from a task executor instead of the calling thread. The method can return void, Future, or CompletableFuture. I'd configure a custom ThreadPoolTaskExecutor bean to control core size, max size, and queue capacity rather than relying on the default.
java@Async
public CompletableFuture<Employee> fetchEmployee(Long id) {
    return CompletableFuture.completedFuture(repo.findById(id).orElseThrow());
}

i. How does Spring handle exceptions
The cleanest way is @RestControllerAdvice — a global exception handler that intercepts exceptions thrown by any controller. I define methods annotated with @ExceptionHandler for each exception type I want to handle, and return a structured error response with the appropriate HTTP status code. This keeps exception handling out of the controller logic and centralizes it in one place.
java@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(EmployeeNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(EmployeeNotFoundException e) {
        return new ErrorResponse(e.getMessage());
    }
}

j. How does Spring validate data
Spring uses the Bean Validation API — I add validation annotations on the fields of my request DTO, like @NotNull, @NotBlank, @Size, @Min, @Max, @Email. Then in the controller I add @Valid on the @RequestBody parameter. If validation fails Spring automatically throws a MethodArgumentNotValidException before the method body even runs. I catch that in my @RestControllerAdvice and return a 400 with details about which fields failed.
javapublic class CreateEmployeeRequest {
    @NotBlank
    private String name;
    @Min(0)
    private int age;
}

@PostMapping
public Employee create(@Valid @RequestBody CreateEmployeeRequest req) {}

k. How does Spring do logging
Spring Boot uses SLF4J as the logging facade with Logback as the default implementation. I inject a logger using LoggerFactory or just annotate the class with Lombok's @Slf4j. Then I log at different levels — debug for development details, info for normal flow, warn for unexpected but recoverable situations, error for failures. Log levels can be configured per package in application.properties so I can turn on debug logs for my own code without flooding the output with framework logs.
java@Slf4j
@Service
public class EmployeeService {
    public Employee getById(Long id) {
        log.info("Fetching employee {}", id);
        return repo.findById(id).orElseThrow();
    }
}

l. Cache hit vs cache miss
A cache hit is when the requested data is found in the cache — response is fast, no database call needed. A cache miss is when the data isn't in the cache — the system falls through to the database, fetches the data, stores it in the cache for next time, then returns it. The goal is to maximize hit rate. In Spring I use @Cacheable on a service method — first call is a miss and hits the database, subsequent calls with the same key are hits and return from cache directly. Redis is the typical cache store for distributed systems.

m. Redis basics
Redis is an in-memory key-value store. Because data lives in memory it's extremely fast — microsecond read and write times. It supports multiple data structures: strings, lists, sets, sorted sets, and hashes. Common use cases are caching to reduce database load, session storage, rate limiting, and pub/sub messaging. Data can optionally be persisted to disk. In Spring Boot I add the spring-boot-starter-data-redis dependency, configure the host in application.properties, and use either RedisTemplate for manual operations or @Cacheable/@CacheEvict annotations for declarative caching.

n. @RestControllerAdvice vs @ControllerAdvice
@ControllerAdvice is the base annotation — it applies globally to all controllers and is typically used in MVC apps that return views. @RestControllerAdvice is @ControllerAdvice plus @ResponseBody, meaning every handler method in it automatically serializes the return value to JSON. For REST APIs I always use @RestControllerAdvice so my error responses get properly serialized without needing to add @ResponseBody to every method.
