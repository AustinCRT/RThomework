Diff between recursion and iteration
Both are ways to repeat logic. Iteration uses loops — straightforward, memory efficient, runs in constant stack space. Recursion is when a function calls itself — more intuitive for problems with naturally recursive structure like tree traversal, but each call adds a frame to the call stack so deep recursion risks a stack overflow. Another approach for problems like sorting or searching is divide and conquer — break the problem into smaller subproblems, solve each one, combine the results. Divide and conquer is typically implemented recursively but can also be done iteratively with an explicit stack. In interviews I'd use recursion for trees and graphs, iteration for simple sequential logic, and divide and conquer for things like merge sort or binary search.

LinkedHashMap vs HashMap
Both HashMap and LinkedHashMap store data as key-value pairs and both give O(1) average for get and put. The difference is insertion order. HashMap makes no guarantee about the order of entries when you iterate — you can get them back in any sequence. LinkedHashMap maintains a doubly linked list internally that tracks insertion order, so iteration always comes back in the order I put things in. I use HashMap when I just need fast key-value lookups and order doesn't matter. LinkedHashMap is my choice when I need predictable iteration order, or for building an LRU cache by overriding removeEldestEntry.

How to group people by age — key: age, value: list of people
I'd use the Stream API. Starting with a List of People, I call stream() then collect with Collectors.groupingBy passing Person::getAge as the classifier. That gives me a Map<Integer, List<People>> where each key is an age and the value is all people with that age. If streams aren't an option I'd do it with a for loop — iterate the list, check if the age key exists in the map, add to the existing list or initialize a new one.

How to write a REST API in Spring Boot
I'd start with a Controller class annotated with @RestController and @RequestMapping to set the base path. Then I define methods for each endpoint — @GetMapping for reads, @PostMapping for creates, @PutMapping or @PatchMapping for updates, @DeleteMapping for deletes. Request body comes in via @RequestBody, path variables via @PathVariable, query params via @RequestParam. The controller calls a Service layer for business logic, the Service calls a Repository which extends JpaRepository for database access. I'd also add a global @ControllerAdvice for exception handling so errors return proper HTTP responses instead of raw 500s.

How did you debug?
My process starts with the logs — I look at the HTTP status code and find the error or exception message to understand what went wrong. Then I take that request and try to reproduce it, first in the prod environment to confirm, then bring the same request payload into dev. I use Postman to replay the exact request, then open IntelliJ and set a breakpoint at the suspected location. Running in debug mode I can step through execution and inspect actual variable values at each point. That flow — log analysis, reproduction, Postman, then IntelliJ breakpoint — covers most issues I'd encounter.

New features in Java 11
Java 11 was a major LTS release with several important additions. The HTTP Client API became a standard feature — a modern async-capable replacement for HttpURLConnection. New String utility methods were added: isBlank, strip, lines, and repeat. ZGC was introduced as an experimental low-latency garbage collector targeting pause times under 10ms. Epsilon GC was also added — a no-op collector that never actually reclaims memory, useful for performance testing where you want zero GC overhead. And single-file Java programs can now be run directly with java without a separate compile step.

Design a locking schema where thread calling method1() must wait until another thread calls method2()
This is a classic thread signaling problem. I'd use wait and notify inside a synchronized block. method1() acquires the lock and calls wait() — that releases the lock and puts the thread in WAITING state. method2() acquires the same lock, does its work, then calls notify() to wake up the waiting thread. I'd wrap the wait() in a while loop to guard against spurious wakeups. A cleaner modern approach is CountDownLatch initialized to 1. method1() calls latch.await() which blocks until the count reaches zero. method2() calls latch.countDown() which releases method1(). CountDownLatch is simpler and more readable than raw wait and notify for this kind of one-time signaling.
