SQL vs NoSQL
SQL databases are relational — data is stored in tables with a fixed schema, and relationships between tables are managed with foreign keys. They're ACID compliant so transactions are reliable. I'd use SQL when my data is structured and relationships matter, like an employee management system. NoSQL databases are non-relational — they store data as documents, key-value pairs, or graphs, with a flexible schema. They scale horizontally more easily and handle unstructured or fast-changing data better. I'd use NoSQL for things like user sessions, product catalogs, or large-scale real-time data. Common examples are PostgreSQL for SQL and MongoDB or Redis for NoSQL.

What is database normalization
Normalization is the process of organizing a database to reduce data redundancy and improve data integrity. It works by splitting data into separate tables and defining relationships between them. The main normal forms are 1NF — no repeating groups, each column holds atomic values. 2NF — no partial dependencies, every non-key column depends on the whole primary key. 3NF — no transitive dependencies, non-key columns depend only on the primary key and nothing else. The goal is to make sure the same data isn't stored in multiple places, so updates only need to happen in one spot.

Vertical scaling vs horizontal scaling
Vertical scaling means adding more resources to one machine — more CPU, more RAM, bigger disk. Simple but has a ceiling, there's only so much hardware I can add to one server. Horizontal scaling means adding more machines and distributing the load across them. More complex to set up but theoretically unlimited. SQL databases traditionally scale vertically. NoSQL databases are designed to scale horizontally which is why they're popular for large-scale applications.

What is ACID
ACID is the set of properties that guarantee database transactions are reliable. Atomicity — a transaction is all or nothing, if one step fails the whole thing rolls back. Consistency — a transaction brings the database from one valid state to another, no constraint violations. Isolation — concurrent transactions don't interfere with each other, each one executes as if it's the only one running. Durability — once a transaction commits, the data is permanently saved even if the system crashes right after. SQL databases like PostgreSQL are fully ACID compliant.

What is CAP
CAP theorem says a distributed system can only guarantee two out of three properties at the same time. Consistency — every read gets the most recent write. Availability — every request gets a response, even if it's not the most recent data. Partition tolerance — the system keeps working even if network partitions cause some nodes to lose communication. In practice partition tolerance is non-negotiable in a distributed system, so the real choice is between consistency and availability. SQL databases typically choose consistency over availability. Many NoSQL databases choose availability and eventual consistency instead.

);

    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    dept_id INT,
    age INT,
    salary DECIMAL(10,2),
    FOREIGN KEY (dept_id) REFERENCES Department(id)
);
