a. What is AOP
AOP stands for Aspect-Oriented Programming. The idea is that in any application, there are certain concerns like logging, security, and transaction management that cut across multiple layers and classes. Instead of repeating that logic everywhere, AOP lets you pull it out into a separate class called an aspect, and then automatically apply it to the methods you care about. So your business logic stays clean and the cross-cutting concerns are handled in one place.

b. What is JoinPoint and Aspect in AOP
A JoinPoint is a specific point during the execution of your program where the aspect can intercept — in Spring AOP, that's always a method execution. It carries information about the method being called, like the method name and its arguments. An Aspect is a class annotated with @Aspect and @Component. It defines two things — where to intercept using @Pointcut, and when to run the logic using advice annotations like @Before, @After, and @Around.

c. What is Aspect, JoinPoint, Pointcut, Advice, Target
Aspect is a class annotated with @Aspect and @Component. It defines where to intercept with @Pointcut and when to run the logic with advice annotations — that's the core of AOP. JoinPoint is the actual method execution point that the aspect intercepts, and it carries the method signature and arguments. Pointcut is the expression that defines where — which methods or which package to target. Advice is the when — it's the actual logic that runs at a specific timing, like @Before runs before the method, @After runs after, @Around wraps the entire execution. Target is the original object whose method is being intercepted by the aspect.

d. ApplicationContext vs BeanFactory
Both are IoC containers in Spring, but ApplicationContext is the more advanced one. BeanFactory is the basic container — it creates beans lazily, meaning it only creates a bean when you first ask for it. ApplicationContext extends BeanFactory and adds more features like eager initialization of singleton beans at startup, event publishing, and AOP integration. In practice we always use ApplicationContext in Spring Boot. BeanFactory is mostly legacy.

e. How is Spring MVC Working Flow
When a request comes in, it first hits the DispatcherServlet, which is the front controller. The DispatcherServlet asks the HandlerMapping to figure out which controller should handle this request based on the URL. Once it finds the right controller, it calls that controller method. The controller processes the request, calls the service layer, gets the result, and returns the response. If it's a REST controller, the response gets serialized to JSON and sent back to the client. If it's a regular controller, the ViewResolver resolves the view name to an actual template and renders it.

f. How does @Autowired work
When Spring starts up and does component scanning, it creates all the beans and registers them in the IoC container. When it sees @Autowired on a field, constructor, or setter, it looks into the container and finds a bean that matches the required type, then injects it automatically. By default it injects by type. If there are multiple beans of the same type, it will throw an error unless you use @Primary or @Qualifier to resolve the conflict.

g. @Autowired and @Qualifier
@Autowired tells Spring to automatically inject a dependency by type. @Qualifier is used together with @Autowired when you have multiple beans of the same type and you need to specify exactly which one you want. For example if you have two implementations of the same interface, you put @Qualifier with the bean name to tell Spring which specific implementation to inject at that injection point.

h. 2 use cases for @PostConstruct and @PreDestroy
@PostConstruct runs right after the bean is created and all dependencies are injected. First use case is loading initial data into a cache so it's ready before the application starts serving requests. Second use case is validating configuration properties right after the bean is fully initialized to make sure everything is set up correctly before the app runs.
@PreDestroy runs just before the bean is destroyed when the application shuts down. First use case is closing open database connections or releasing resources to prevent memory leaks. Second use case is saving in-memory state to a database before the application goes down so no data is lost.
