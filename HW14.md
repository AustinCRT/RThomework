Conceptual Questions

1. What is Index

An index is a special data structure that the database creates on top of one or more columns in a table. It allows the database to find rows much faster without scanning every single row. Without an index, the database has to do a full table scan, which is O(n). With an index like a B+ Tree, the search becomes O(log n). The trade-off is that every write operation (INSERT, UPDATE, DELETE) needs to update the index as well, so write performance slows down. That's why we don't just add indexes to every column — we only index columns that are frequently queried.


2. Clustered Index vs Non-Clustered Index

A clustered index determines the physical order that data rows are stored on disk. Each table can only have one clustered index, and by default it is the primary key. When you query by primary key, the database finds the row directly because the data is physically sorted that way.

A non-clustered index is a separate structure that stores the indexed column values and a pointer back to the actual data row. A table can have multiple non-clustered indexes. When you query by a non-clustered index, the database first looks up the index, then follows the pointer to fetch the actual row.


3. Data Structure for Index

The most common data structures used for indexes in SQL databases are:


B+ Tree — the default in InnoDB (MySQL) and PostgreSQL. It supports both equality and range queries efficiently. Leaf nodes are linked horizontally, so range scans like WHERE salary BETWEEN 50000 AND 90000 are very fast.
B Tree — similar to B+ Tree but data is stored at all levels, not just leaf nodes. Less efficient for range queries.
Hash Table — extremely fast for equality queries (WHERE id = 101), but cannot support range queries at all.
R Tree — used for spatial/geographic data.
k-d Tree — used for multi-dimensional data.
Bitmap — used for low-cardinality columns like gender or status.


In practice, B+ Tree is the go-to choice because it handles both point lookups and range queries well.


4. View vs Stored Procedure

A View is a saved SELECT query. It does not store data itself — every time you query a view, it runs the underlying SELECT statement. It is used to simplify complex queries and provide a clean interface to the data.

sqlCREATE VIEW it_employees AS
SELECT name, salary FROM EMPLOYEE
JOIN DEPARTMENT d ON EMPLOYEE.dept = d.dept_key
WHERE d.name = 'IT';

-- Usage
SELECT * FROM it_employees;

A Stored Procedure is a saved block of SQL logic that can include control flow like IF/ELSE and loops, accept input parameters, and execute multiple statements. It is used to encapsulate business logic inside the database.

sqlCREATE PROCEDURE get_employees_by_dept(IN dept_name TEXT)
LANGUAGE plpgsql AS $$
BEGIN
  SELECT e.name, e.salary
  FROM EMPLOYEE e
  JOIN DEPARTMENT d ON e.dept = d.dept_key
  WHERE d.name = dept_name;
END;
$$;

-- Usage
CALL get_employees_by_dept('IT');

The key difference is that a View is read-only and just wraps a query, while a Stored Procedure can contain complex logic, parameters, and multiple operations.


5. View vs Materialized View

A regular View is virtual — it runs the underlying query every time you access it. If the underlying query is expensive, the view is also slow.

A Materialized View physically stores the query result on disk. It does not re-execute the query on every access, so reads are much faster. The trade-off is that the data can become stale — you need to refresh it manually or on a schedule.

sql-- Create materialized view
CREATE MATERIALIZED VIEW dept_avg_salary AS
SELECT dept, AVG(salary) AS avg_salary
FROM EMPLOYEE
GROUP BY dept;

-- Refresh when data changes
REFRESH MATERIALIZED VIEW dept_avg_salary;

Use a regular View when data freshness is critical. Use a Materialized View when the query is expensive and you can tolerate slightly stale data (e.g. a dashboard that refreshes every hour).


6. How to Tune a SQL Query

When a SQL query runs slow, I follow this process:


Run EXPLAIN to analyze the execution plan and identify where the bottleneck is — whether it is a full table scan, a missing index, or an expensive JOIN.
Reduce unnecessary JOINs — remove any JOINs that are no longer needed due to business logic changes.
Add indexes on columns used in WHERE, JOIN ON, and ORDER BY clauses. Make sure the execution plan shows index scans, not sequential scans.
Use UNION ALL instead of UNION — UNION removes duplicates which requires sorting. If duplicates are acceptable, UNION ALL is faster.
Select only the fields you need instead of SELECT * — this reduces the amount of data read from disk.
Use JOIN ON instead of WHERE for joining tables — it is cleaner and the optimizer handles it better.
Avoid LIKE '%keyword%' with a leading wildcard — this disables index usage and forces a full table scan. LIKE 'keyword%' can still use an index.
Use LIMIT for pagination on large datasets instead of loading everything at once.
Replace subqueries with window functions where possible — DENSE_RANK() is often faster than nested subqueries for ranking problems.
Avoid IN with large lists — this can trigger a full table scan. Use EXISTS or JOIN as alternatives.
